### Note on `-Wpedantic` for this environment

The platform env is the first consumer to `#include` ESP-IDF headers from C++
(`esp_timer.h`, `esp_log.h`, `lwip/sockets.h`). ESP-IDF's `esp_libc` platform
headers and lwIP's socket shim use the GCC `#include_next` extension, which
`-Wpedantic` flags. The `esp32-s3-platform` env keeps the full strict flag set
(`-Wall -Wextra -Wpedantic -Werror`) but adds `-Wno-error=pedantic` scoped to
that one environment in `src/CMakeLists.txt`, so the ESP-IDF system-header
extension surfaces as a non-fatal warning without weakening the strict set for
project-owned code in any other image.

## What this evidence does and does not establish

- The Phase 5.2 platform-esp32 package compiled and linked for ESP32-S3; the
  figures above are compile-only image evidence.
- No firmware upload, radio transmit, WiFi/netif bring-up, runtime timing,
  stack high-water mark, heap measurement, or physical-hardware behavior was
  performed or recorded.
- The `app_main` example binds a dedicated-server `TNetHost` over a real lwIP
  socket but initializes no netif/WiFi, so no UDP datagram can actually flow;
  the image proves composition and compilation of the full stack, not that any
  packet reaches the wire. The exact oversize-datagram receive behavior of
  `FEsp32UdpDriver::TryReceive` is unverified at runtime (the sizing peek uses
  `MSG_PEEK` + `MSG_TRUNC` when lwIP defines it, otherwise `MSG_PEEK` alone into
  a 1200-byte internal scratch; see `src/Esp32SocketGlue.h`).
