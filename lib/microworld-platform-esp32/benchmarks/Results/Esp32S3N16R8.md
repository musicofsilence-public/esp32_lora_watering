## Verified result

The new `esp32-s3-platform` consumer environment — the first ESP32 image that
composes the full stack (`FEsp32TimeSource` + `FEsp32UdpDriver` + `TNetHost` +
`TEngineHost` over the `TNetHostFrame`/`INetworkFrame` seam) — was built with
PlatformIO. Build, library-dependency, and cache directories were redirected to

repository.

```powershell
$env:PLATFORMIO_BUILD_DIR   = '<temporary-evidence-root>\pio-build'
$env:PLATFORMIO_LIBDEPS_DIR = '<temporary-evidence-root>\pio-libdeps'
$env:PLATFORMIO_CACHE_DIR   = '<temporary-evidence-root>\pio-cache'
pio run -d lib/microworld/tests/consumer -e esp32-s3-platform
```

| Resource | Used | Total | Percent |
| --- | ---: | ---: | ---: |
| RAM | 21,772 bytes | 327,680 bytes | 6.6% |
| Flash | 283,473 bytes | 4,194,304 bytes | 6.8% |

The pre-existing `esp32-s3-engine` and `esp32-s3-net` consumer environments

### Note on `-Wpedantic` for this environment

The platform env is the first consumer to `#include` ESP-IDF headers from C++
(`esp_timer.h`, `esp_log.h`, `lwip/sockets.h`). ESP-IDF's `esp_libc` platform
headers and lwIP's socket shim use the GCC `#include_next` extension, which
`-Wpedantic` flags. The `esp32-s3-platform` env keeps the full strict flag set
(`-Wall -Wextra -Wpedantic -Werror`) but adds `-Wno-error=pedantic` scoped to
that one environment in `src/CMakeLists.txt`, so the ESP-IDF system-header
extension surfaces as a non-fatal warning without weakening the strict set for
project-owned code in any other image.

## What this evidence does and does not establish

- The Phase 5.2 platform-esp32 package compiled and linked for ESP32-S3; the
  figures above are compile-only image evidence.
- No firmware upload, radio transmit, WiFi/netif bring-up, runtime timing,
  stack high-water mark, heap measurement, or physical-hardware behavior was
  performed or recorded.
- The `app_main` example binds a dedicated-server `TNetHost` over a real lwIP
  socket but initializes no netif/WiFi, so no UDP datagram can actually flow;
  the image proves composition and compilation of the full stack, not that any
  packet reaches the wire. The exact oversize-datagram receive behavior of
  `FEsp32UdpDriver::TryReceive` is unverified at runtime (the sizing peek uses
  `MSG_PEEK` + `MSG_TRUNC` when lwIP defines it, otherwise `MSG_PEEK` alone into
  a 1200-byte internal scratch; see `src/Esp32SocketGlue.h`).

# Phase 6.2 runtime margins (measured on hardware)

Status: **Measured on physical ESP32-S3 on 2026-07-21 under explicit human
authorization to flash.** The `esp32-s3-benchmark` image was flashed to a
connected board and the labeled serial lines below were captured over USB.

## Environment

| Tool / setting | Value |
| --- | --- |
| Board | `esp32-s3-devkitc-1` (ESP32-S3, USB VID:PID `303A:1001`), 8 MB PSRAM |
| CPU frequency | 160 MHz (observed in boot log) |
| ESP-IDF | 6.0.1 |
| Xtensa-ESP-ELF GCC | 15.2.0 |
| Environment | `esp32-s3-benchmark` (release, `-Os`, strict set + scoped `-Wno-error=pedantic`) |
| Image | RAM 43,148 B (13.2% of 327,680); Flash 313,269 B (7.5% of 4,194,304) |
| Harness | `lib/microworld/tests/consumer/src/Esp32BenchmarkMain.cpp` |

Capture commands (PIO dirs redirected to a temporary evidence root so no
generated file entered the repository):

```powershell
pio run -d lib/microworld/tests/consumer -e esp32-s3-benchmark -t upload --upload-port COM5
pio device monitor -d lib/microworld/tests/consumer -e esp32-s3-benchmark
```

## Measured result

Representative world: 8 actors / 16 components (two per actor) / 8 looping
timers, all spawned during setup before `BeginPlay`.

| Measurement | Result | Notes |
| --- | --- | ---: |
| Tick duration (1000 iterations) | min 62 µs / mean 73 µs / max 114 µs | steady-state; each tick completes one full GC cycle (budget sweep 32 ≥ 32 objects) |
| GC pause per Advance slice | min 21 µs / mean 25 µs / max 39 µs | standalone probe, budget `{root=1,mark=1,sweep=8}`, 4 slices/cycle over 32 slots |
| Net pump (no traffic) | mean 47 µs / 1000 iterations | `PumpReceive`+`PumpSend` overhead only; a live datagram cost needs a peer and is out of scope |
| Heap consumed by world setup | 580 bytes | fixed caller-owned engine storage keeps world setup near zero heap |
| Main-task stack free after setup | 2,476 B of 3,584 | high-water headroom after the composition moved to static storage |

Raw serial (`mwbench` tag):

## What this evidence does and does not establish

- The full managed stack (`TEngineHost` + `TNetHost` over the network-frame
  seam, `FEsp32TimeSource`, `FEsp32UdpDriver`) runs on real ESP32-S3 hardware
  and completes bounded tick / GC / net-pump work at the margins above.
- Timings are at 160 MHz; a 240 MHz build would be roughly proportionally
  faster. Numbers are single-run captures, not statistical distributions.
- Net pump is the no-traffic polling overhead only — no netif carries a real
  datagram, so per-packet cost on the wire is still unmeasured.
- Two runtime defects had to be fixed for the harness to run at all (both
  invisible to the compile-only Part A proof): the harness never initialized
  the lwIP TCP/IP stack before opening a socket (`esp_netif_init` +
  `esp_event_loop_create_default` added), and the composition objects were
  stack locals that overflowed the 3,584-byte main task stack (moved to static
  `.bss`). See the MICROWORLD_ROADMAP.md §6 decision record.
